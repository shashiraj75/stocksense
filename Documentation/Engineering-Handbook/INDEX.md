## System Design Specifications (SSDS)

- [SSDS-000 — StockSense360 System Architecture](SSDS/SSDS-000-StockSense360-System-Architecture.md) — the master architecture document; the single source of truth for how every component of the platform fits together. Read this before designing anything that spans more than one engine or service.
- [SSDS-001 — Feature Specification Template](SSDS/SSDS-001-Feature-Specification-Template.md) — for any user-facing feature or material behavior change.
- [SSDS-002 — Sprint Specification Template](SSDS/SSDS-002-Sprint-Specification-Template.md) — for scoping a numbered engineering sprint against the Roadmap.
- [SSDS-003 — StockSense360 Business Quality Engine](SSDS/SSDS-003-StockSense360-Business-Quality-Engine.md) — full design spec (framework, metrics, sector adaptation, output contract, validation strategy) for the engine answering "is this fundamentally an outstanding business worthy of long-term ownership?" Implemented in Sprint #004 (`backend/services/business_quality_engine.py`), validated and calibrated across Sprints #004a, and integrated into its first consumer (Multibagger Quality Compounder, US-only) in [Sprint #005](Releases/Sprint-005-Business-Quality-Engine-Multibagger-Integration.md).
- [SSDS-004 — StockSense360 India Fundamentals Data Strategy](SSDS/SSDS-004-StockSense360-India-Fundamentals-Data-Strategy.md) — strategy proposal identifying two unverified data hypotheses as the highest-leverage next step before any provider decision. **Both hypotheses verified — see the validation study below.**
- [SSDS-005 — StockSense360 Financial Strength Intelligence Engine](SSDS/SSDS-005-StockSense360-Financial-Strength-Intelligence-Engine.md) — full design spec (scope boundary vs. Business Quality Intelligence, scoring categories, Financial Stress Simulation framework, data requirements, sector adaptations, confidence model, output contract, validation strategy, future sprint roadmap) for the engine answering "could this company survive a downturn and service its obligations?" Formalizes the [Financial Strength Intelligence Design Study](Architecture/Financial-Strength-Intelligence-Design-Study.md) into a binding specification. **Implemented, validated, calibrated, and integrated — see [EPIC-002 Closure](EPICS/EPIC-002-Financial-Strength-Intelligence-Closure.md)** for the full Sprints #001–#012 record; this document remains the governing spec for the engine's category/threshold design.
- [India Fundamentals Data Validation & Derivation Study](Architecture/India-Fundamentals-Data-Validation-Study.md) — 65-company live validation of SSDS-004's hypotheses. **Result: the existing screener.in pipeline already supports 4 of 5 targeted Business Quality Engine metrics (Altman, Sloan, Cash Conversion, Asset Turnover) at 97–100% completeness** — the Total-Assets-via-balance-sheet-identity hypothesis is proven (97% cross-check match against an independent source, both outliers individually explained). Only Beneish M-Score is a confirmed, total gap (0/65). **Recommends no new data provider and Sprint #007 implement a narrowly-scoped India Business Quality Adapter** (the 4 proven metrics only) — not further research, not an NSE/paid-vendor build.
- [SSDS-006 — StockSense360 Data Fabric & Provider Architecture](SSDS/SSDS-006-StockSense360-Data-Fabric-Provider-Architecture.md) — the permanent, provider-agnostic foundation every future data source must be built against: a 7-layer pipeline (Provider Adapter → Normalization → Validation → Confidence → Data Fabric resolution → Cache → Engines) with a standing Provider Adapter Standard, a field-normalization schema, per-field provenance tagging, a provider-confidence model kept explicitly separate from engine-level confidence, failover rules, and a multi-provider conflict-resolution policy. Formalizes ideas the Data Independence & Provider Strategy report only proposed informally into a binding spec. **Its first provider (SEC EDGAR) and first consumer engine (Financial Strength) are now implemented and live — see [EPIC-002 Closure](EPICS/EPIC-002-Financial-Strength-Intelligence-Closure.md).** This document remains the governing spec for the Fabric's layered architecture; the Provider Registry/Normalization/Confidence layers as standalone modules (Section 15) remain future work.

## Engineering Audits

- [SEAR-001 — Selection Engine Engineering Audit](Architecture/Sprint-001-Selection-Engine-Audit.md) — the founding audit; source of the Roadmap's phases and most named risks/gaps referenced throughout the standards above.
- [Mini Security Audit — Pre-Epic 003](Architecture/Security-Audit-Pre-Epic-003.md) — a platform-wide security review (auth, authorization, secrets, API, frontend, providers, infra, dependencies, OWASP Top 10). **Found one Critical finding: every user-data API route (Portfolio, Watchlist, Alerts, Terms-Acceptance) accepts a `user_id` with zero authentication or authorization enforcement** — confirmed exploitable end-to-end (frontend sends the real Supabase user ID with no token attached; the backend's DB role bypasses RLS, so there's no defense-in-depth underneath either). One High finding (a CORS policy that would become directly exploitable if a cookie-based fix to the Critical finding is added without also tightening it). Security score 4/10; recommends blocking any Private Beta involving real portfolio/watchlist/alert data until fixed, while confirming Epic 003 can proceed on schedule (the finding is isolated to the user-account-data layer, not the intelligence engines). No code modified — audit only, fixes not yet applied. **Both findings fixed in [Security Remediation Sprint #001](Architecture/Security-Remediation-Sprint-001.md).**
- [Security Remediation Sprint #001](Architecture/Security-Remediation-Sprint-001.md) — fixes the audit above's C-1 (Broken Access Control/IDOR) and H-1 (permissive CORS) findings. Added real Supabase JWT verification and ownership checks (`services/auth.py`) to Portfolio, Watchlist, Alerts, and Terms-Acceptance; tightened CORS to explicit approved origins only, removing the `*.vercel.app` wildcard; added per-IP rate limiting (`services/rate_limit.py`); wired the frontend's shared API client to attach the Supabase access token. 58 new regression tests, sanity-checked by reverting the fix and confirming 16/20 portfolio tests fail without it. 500/500 full suite passing. Paper Trading's pseudonymous session ID (the audit's L-1, a different and lower-priority risk class) deliberately left out of scope. **Includes a follow-up fixing a production-only defect: Supabase's ES256 JWT Signing Keys weren't supported, rejecting every legitimate token with 401 — fixed via JWKS-based verification alongside the legacy HS256 path. 510/510 passing.**
- [Production Validation Sprint — Final Gate Before Epic 003](Architecture/Production-Validation-Sprint-Pre-Epic-003.md) — forensic validation of both security sprints above, confirming zero regressions to Portfolio/Watchlist/Alerts/Terms or the Prediction Engine/intelligence layer (proven by direct git-diff forensics, not assumed). Reproduced every named attack vector (forged/modified/expired JWT, wrong audience/issuer/signature, algorithm confusion, missing/malformed headers) — all blocked except one: **JWT `iss` claim is never validated (Low/Medium, low practical exploitability)**. Also found the rate limiter likely buckets all users together behind Railway's reverse proxy since `get_remote_address` doesn't check `X-Forwarded-For` the way `auth.py`'s pre-existing IP-logging code already does (**Medium, reliability risk, not a security hole**). Four pre-existing, unrelated Low findings named for completeness (Portfolio duplicate prevention, symbol validation, missing FK constraints, 200-not-404 on no-op updates). 524/524 tests passing. Explicit scope limitations named: no live browser/production-URL/database access this session. **Verdict: Ready after minor fixes** — neither finding blocks Epic 003.
- [Business Quality Engine — Production Readiness Validation](Architecture/Business-Quality-Engine-Production-Readiness-Validation.md) — the founding 55-company live validation. Found Altman Z-Score/Sloan Accruals never computed against live data and Piotroski had no sector-awareness — both fixed in [Sprint-004a](Releases/Sprint-004a-Business-Quality-Engine-Calibration.md). That fix surfaced a follow-up false positive (HON/ORCL hard-rejected) — root-caused (Altman's X1/X2/X3 numerator terms had no fallback, only the denominator did) and fixed, commit `ffaafcb`.
- [Business Quality Engine — Final Production-Readiness Re-Validation](Architecture/Business-Quality-Engine-Final-Production-Readiness-Revalidation.md) — full Phase 1–9 re-validation after all three fixes, on a fresh live run of the same 53-company dataset. **Verdict: production-ready for the scope validated.** Altman/Accruals availability 100%, HON/ORCL false rejections confirmed fixed, genuine distress (IDEA/LCID/RIVN/PTON) still correctly rejected, no new false positives/negatives.
- [Sprint #005 — Business Quality Engine → Multibagger Integration](Releases/Sprint-005-Business-Quality-Engine-Multibagger-Integration.md) — the recommended first consumer, integrated. **US-only** by evidence-based necessity (the IN nightly refresh sources from screener.in and has no yfinance `Ticker` in scope; adding one would be a broad refactor out of scope) — named explicitly, not silently scoped down. Promotion-only and red-flag-only additions to the scorecard; both pre-existing golden tests pass unchanged. 194/194 tests passing, GitHub Actions green.
- [Sprint #007 — India Business Quality Adapter](Releases/Sprint-007-India-Business-Quality-Adapter.md) — closes Sprint #005's IN gap. A new `india_business_quality_adapter.py` maps screener.in data into the engine's input model (using Sprint #006's proven Total-Assets-via-identity derivation) and is wired into the IN nightly refresh — populating Business Quality scores for India through the unmodified engine, no new provider. **Result on the full 65-company validated universe: 0 errors, 0 rejections, Altman/Sloan/Cash-Conversion all 100%, stable 91.7% confidence.** A genuine classifier defect was found and fixed along the way (screener.in's real Indian sector labels — "Fast Moving Consumer Goods", "Packaged Foods", "Power - Transmission" — fell through to `OTHER`). 262/262 tests passing. The Business Quality Engine now populates scores for **both** markets; next consumer step is wiring the IN fields into the Multibagger scorecard (already market-agnostic since Sprint #005).

## How to use this handbook

1. Before starting any implementation task, read SES-001–005 if you haven't already internalized them this session.
2. Check the [Product Glossary](Glossary/StockSense360-Product-Glossary.md) before naming any engine, score, or feature — use the name it specifies, don't invent a new one.
3. For anything spanning more than one engine or service, read [SSDS-000](SSDS/SSDS-000-StockSense360-System-Architecture.md) first. Before starting a feature or a sprint, fill out the matching SSDS-001/002 template.
4. While working, follow the relevant SES standard for the kind of change you're making.
5. When you finish a sprint, write its report under `Releases/` per SES-004 §3, and update this INDEX if you added a new governing document.
